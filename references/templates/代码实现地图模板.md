# 代码实现地图模板

## 资深角色视角

- 主程/架构师：输出系统边界、数据流和状态机，而不是粘贴可复用源码片段。
- 逆向分析者：区分源码、IL、IL2CPP/native、YYC、Blueprint/GDScript/GML 的可读性和证据等级。
- 工具工程师：记录工具版本、导出失败、地址映射和可复现路径。
- 合规边界：代码分析只服务原创化学习；如涉及调色板、形状语言或制作约束，只记录系统如何支撑这些抽象规则。

## 引擎分支

| 引擎/工具 | 代码地图重点 |
|---|---|
| Unity | Assembly-CSharp、Managed IL、IL2CPP DummyDll、AnimatorController、ScriptableObject、Addressables key |
| Unreal | Blueprint asset、C++ 类名线索、Actor/Component、AssetRegistry、Gameplay tags/data assets |
| GameMaker | Object/Event、GML 脚本、YYC native 边界、room/object/sprite 关系 |
| Godot | `.gd`、`.tscn` node script、autoload、signal、resource 引用 |
| 未知引擎 | 字符串表、导入表、配置脚本、资源引用、日志函数名、工具可读性 |

## 必填分析细节大纲

1. 可读性边界：源码/脚本/IL/native 是否可读，为什么。
2. 主循环和生命周期：启动、加载、更新、输入、物理、动画、渲染、卸载。
3. 核心系统索引：玩家、敌人、战斗、道具、UI、存档、关卡、音频。
4. 类型/对象/节点/Actor 表：名称、来源、职责、字段、方法、证据。
5. 数据流：配置表到运行时对象，资源引用到实例，事件到反馈。
6. 状态机：角色动作、战斗状态、UI 状态、关卡状态。
7. 行为伪代码：只写抽象流程，不复制可复用源码片段。
8. 置信度：观察事实、工具来源、我的判断、待验证假设。
