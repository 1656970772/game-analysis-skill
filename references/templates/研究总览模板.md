# 研究总览模板

## 资深角色视角

- 制作人/架构师：先界定研究目标、授权边界、版本与平台，避免后续报告变成无范围的素材堆。
- 游戏设计负责人：用一句话定义核心体验，再检查技术、美术、关卡、动画是否共同服务这个体验。
- 主美/概念设计师：记录最强视觉识别点和不可复制元素，只把参考转译为抽象原则。
- 合规边界：即使是总览，也要标明原创化方向、调色板/形状语言/制作约束的待验证项。

## 引擎分支

| 引擎/工具 | 总览必须补充 |
|---|---|
| Unity | `*_Data`、脚本后端、Managed/IL2CPP、Addressables/AssetBundle、Scene/Prefab/Animator 线索 |
| Unreal | `.pak/.ucas/.utoc`、mount point、Chunk、AssetRegistry、Blueprint/C++ 边界 |
| GameMaker | `data.win`、texture page、sprite/object/room、GML/YYC 边界、audio group |
| Godot | `project.godot`、`.tscn/.tres/.gd`、scene tree、autoload、signals、resource path |
| 未知引擎 | 文件树、魔数、manifest、可执行文件名、配置、日志、工具候选和待验证假设 |

## 必填分析细节大纲

1. 研究对象：游戏名、版本、平台、构建目录、输入路径、输出路径。
2. 授权边界：用户授权路径、只读/导出范围、不可上传/发布/复用内容。
3. 研究模式：`dry-run`、`inventory`、`art-study`、`tech-study`、`full-study`。
4. 核心体验一句话：玩家在 30 秒内反复做什么，游戏希望制造什么感受。
5. 引擎与工具链判断：观察事实、来源路径、工具输出、置信度、待验证假设。
6. 系统关系摘要：玩法、技术、美术、场景、动画、UI 如何互相支撑。
7. 最强可学习点：技术、管线、玩法、美术、关卡、手感各 1-3 条。
8. 不可复制元素：角色、logo、UI 图标、tileset、关卡布局、源码、原始资源。
9. 本次产物目录：0-4 目录、最终文档、画廊/EXE、阻塞项。
