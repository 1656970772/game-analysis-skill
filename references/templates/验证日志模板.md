# 验证日志模板

## 资深角色视角

- 构建/工具工程师：每个工具命令必须可复现，失败也要保留证据。
- QA/技术负责人：检查源目录是否未修改、输出是否在 study-root、每个分支是否有验收状态。
- 安全/合规审查者：记录工具来源、许可证、哈希和不使用破解/绕过工具的理由。
- 我的判断：验证日志也要覆盖原创化、调色板、形状语言、可读性、制作约束和待验证项是否被最终文档记录。

## 引擎分支

| 引擎/工具 | 验证日志重点 |
|---|---|
| Unity | Unity 本地工具链、性能采样、代码可读性工具、Animator/Scene 分析 |
| Unreal | Unreal 包体只读工具链、性能采样、包体哈希、mount point、chunk |
| GameMaker | UTMT、normalize、organize、Profiler/Debugger、YYC 边界 |
| Godot | pck/project 识别、scene/resource/script 检查、工具候选 |
| 未知引擎 | 魔数、工具下载、`--help`、小样本验证、阻塞项 |

## 必填分析细节大纲

1. 环境：日期、系统、Python/Node/工具版本、工作目录。
2. 授权：输入路径、输出路径、是否只读、是否修改源目录。
3. 命令日志：命令、开始/结束时间、退出码、stdout/stderr 摘要。
4. 工具记录：来源 URL、版本、许可证、哈希、用途。
5. 输出验证：目录、文件数、索引非空、报告生成、画廊/EXE 自测。
6. 失败和阻塞：错误信息、已尝试方案、下一步。
7. 证据索引：截图、CSV/JSON、manifest、hash、报告引用路径。
