# 工具自扩展与使用后迭代

## 适用场景

遇到以下情况时读取本文件：

- `detect_engine.py` 与人工检查无法命中 Unity、Unreal、GameMaker 等已写入分支。
- 命中 Godot、RPG Maker、Ren'Py、MonoGame、HTML5、自研引擎、移动端包体或其他未写明工作流。
- 已有工具无法读取资源包、metadata、纹理、音频、模型、脚本或场景文件。
- 任务中出现新的错误信息、版本兼容问题、GUI 限制、路径编码问题或大文件处理问题。
- 任务结束前需要把本次经验沉淀回 Skill。

## 未知引擎处理流程

1. **只读识别**
   - 记录目录树摘要、最大文件、可执行文件名、manifest/config、日志、包扩展名、文件头魔数和疑似版本。
   - 只读取 metadata、小文本和文件名；不要先批量解包或转换。
   - 把结论分为“观察到的事实”“我的判断”“待验证假设”。

2. **工具发现**
   - 优先顺序：官方导出/调试工具、官方文档、开源 GitHub Release、包管理器、长期维护的社区工具。
   - 搜索目标应包含：引擎名、包扩展名、版本号、`extract`、`unpack`、`viewer`、`asset`、`modding`、`decompiler`、`metadata`。
   - 不使用来源不明的二进制、不下载破解工具、不绕过 DRM/反作弊/联网验证/访问控制。

3. **下载与记录**
   - 可复用工具下载到 Skill 的 `tools/<tool-name>_<version>/`。
   - 一次性、大体积、许可不宜复用或只对当前项目有意义的工具下载到 `<study-root>/4.临时目录/tools/`。
   - 在 `4.临时目录/工具获取记录.md` 记录：工具名、版本、来源 URL、许可证、哈希、下载时间、用途、是否放入 Skill、`--help` 或版本输出。

4. **小样本验证**
   - 先运行 `--help`、`list`、`info`、`dry-run` 或对单个小文件测试。
   - 验证输出路径在 study-root 内，源游戏目录未修改。
   - 记录失败信息；失败也是有效产出，不能把工具失败改写成项目不存在相关内容。

5. **生成临时工作流**
   - 在 `4.临时目录/未知引擎工作流草案.md` 写：检测特征、推荐工具、命令、输出结构、失败边界、下一步。
   - 如果工作流稳定，转写为 `references/<engine-or-format>-workflow.md`。
   - 如果需要脚本化且会重复使用，新增 `scripts/<purpose>.py` 或 `.ps1`，并添加最小测试。

## 常见未覆盖分支提示

| 线索 | 初始判断 | 下一步 |
|---|---|---|
| `project.godot`、`.pck` | Godot | 先查 Godot 官方 pack 格式和版本；优先只读列包与小样本导出 |
| `Game.rgss*`、`Data/*.rvdata*` | RPG Maker | 识别 VX/VX Ace/MV/MZ 分支，先做数据表和图片目录索引 |
| `.rpa`、`.rpyc`、`renpy/` | Ren'Py | 先识别 archive、脚本和图片结构，输出叙事/UI/资源索引 |
| `.xnb` | XNA/MonoGame | 查找 XNB viewer/extractor，只做本机索引和抽象规范 |
| `www/`、`index.html`、`*.wasm` | HTML5/WebGL | 记录构建框架、资源 manifest、bundle、WASM 边界 |
| Android `.apk`、iOS `.ipa` | 移动端包体 | 先做包结构和 manifest 只读分析；不要绕过签名、加密或服务端校验 |
| 未知大包、无扩展大文件 | 自研或加密包 | 记录魔数、熵、字符串、路径线索；只能在合法工具和授权范围内继续 |

## 使用后迭代流程

每次任务结束前执行：

1. 写 `2.报告/使用后复盘与 Skill 迭代记录.md`。
2. 判断是否需要更新 Skill：
   - 新引擎、包格式、工具、错误信息、路径问题、输出模板缺口：必须更新。
   - 只是某个项目的私有内容、原始素材、密钥、商业机密：不得写入 Skill。
3. 更新内容优先级：
   - `test-prompts.json`：新增可复现的触发场景和期望行为。
   - `SKILL.md`：新增通用规则、模式、完成检查或 reference 导航。
   - `references/*.md`：新增详细工作流、工具边界、输出模板。
   - `scripts/` 与 `tests/`：沉淀可重复、易错、需要稳定执行的逻辑。
   - `references/tooling-manifest.json`：记录可复用工具的来源、版本、用途和许可证。
4. 运行最小验证：
   - 文档契约测试。
   - 新增脚本测试。
   - 相关 workflow 的 dry-run 或小样本测试。
5. 在最终回复里说明：本次使用后已如何迭代 Skill；若没有迭代，说明原因和具体待办。
